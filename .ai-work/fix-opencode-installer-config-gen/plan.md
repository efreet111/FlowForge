## 1. Summary

Este plan entrega un instalador FlowForge que **genera** (en lugar de copiar) un `opencode.json` canónico, PII-free, free-Zen-only, con `model-assignments.md` regenerado desde el bloque `provider`, y paridad byte-idéntica entre los installers bash y C#. Cubre los 10 FR + 7 NFR del spec, organizados en 6 capas (Templates / C# / Bash parity / Tests / Doctor / Docs) con DAG de tareas T-001..T-038 y gate de paridad en CI. Single source of truth: `ide/opencode/templates/` consumido por ambos installers vía un manifest `agent-models.json`.

## 5. Testing strategy

### Unit (xUnit + NSubstitute — existing pattern)
- `OpenCodeConfigGenerator`: fresh install, reinstall con custom mcp, reinstall con paid provider preserved (FR-007 Scenario B), `--force-free` downgrade.
- `ModelAssignmentsGenerator`: tabla 9 filas, regex `^opencode-zen/<model>$`, footer comment, cero `claude-`/`gpt-`.
- `PiiScanner`: 4 patrones bloqueantes, permite `$HOME`/`$USER`.
- `ManagedPathsSidecar`: write/read roundtrip, legacy migration (no sidecar → `["mcp.engram"]`).
- `DoctorCommand`: missing `agent` section, PII detection, stale `model-assignments.md`.
- `AgentFrontmatterPatcher`: fresh copy, reinstall sobre frontmatter stale, body preservado.

### Parity (CI gate)
`scripts/test-parity-opencode.sh`:
1. Setup dos temp `$HOME` (`$HOME_A`, `$HOME_B`), mismo `$FLOWFORGE_REPO`.
2. `$HOME_A`: `dotnet run --project src/FlowForge.Installer -- install --ide opencode --yes`.
3. `$HOME_B`: `bash ide/install.sh`.
4. Canonizar: `jq -S '. | del(.["$schema"])' $HOME_A/.config/opencode/opencode.json` idem B.
5. `diff -u A B` — debe ser vacío (modulo timestamp en comment).
6. Idem para `model-assignments.md` y `agents/forge-arch.md` (frontmatter).
7. Exit 0 si diff vacío, 1 si no. Block merge on failure.

### Integration
- `flowforge install --ide opencode --yes` en temp HOME (`XDG_CONFIG_HOME` override), luego `flowforge doctor` → exit 0.
- Re-correr install → idempotente, backup nuevo pero `opencode.json` byte-identical (modulo timestamp).

### Property-based (optional, low priority)
- Idempotencia: 2 installs sucesivos → byte-identical output (modulo timestamp en backup comment).
