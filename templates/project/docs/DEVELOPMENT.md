## Branching strategy

| Branch | Purpose |
|--------|---------|
| `main` | Production-ready code |
| `dev` | Integration branch |
| `feat/{slug}` | Feature branches (one per HU) |
| `fix/{slug}` | Bug fix branches |

Feature branches are created from `dev` and merged via PR. No direct pushes to `main`.

## Code documentation policy

Write inline comments **only** when the code cannot speak for itself.

| Write | Do not write |
|-------|-------------|
| Comment explaining a non-obvious business rule | Comment that repeats the function signature |
| Comment on a known limitation or workaround | Comment on a self-explanatory one-liner |
| Public API `///` / JSDoc with rules not in the signature | Mirror of parameter types already in the signature |

> Full policy: [FlowForge ADR-002 — Scaffold documentation policy](https://github.com/efreet111/FlowForge/blob/main/docs/decisions/ADR-002-scaffold-doc-policy.md)

## Adding a new feature (FlowForge workflow)

```
/flow-start {slug}      → Discovery + spec.md (CKP-1: human approves)
/flow-plan              → plan.md (CKP-2: human green-lights)
/flow-dev               → code + tests
/flow-verify            → audit (CKP-3: max 3 rework cycles)
/flow-close             → session close + HU updated (CKP-4: deploy decision)
```

Each feature lives in `.ai-work/{slug}/` during development. After close, the folder is archived.
